# Bookings Module PRD

## 業務目的

`bookings` 模組負責場地預約申請、個人申請查詢、預約修改與撤回、場地日曆占用查詢，以及管理端審核預約。模組核心目標是確保同一場地、同一日期、同一時段不會同時存在多筆已通過預約。

## 使用者角色

- 一般使用者：建立預約、查詢自己的預約、修改可修改狀態的預約、撤回自己的預約、查看日曆占用狀況。
- 管理員：查詢預約列表與詳情、審核通過、調整審核狀態、軟刪除預約案。
- 公開查詢：部分日曆與已通過預約查詢 endpoint 目前未在 controller 標註 `SecurityRequirement`，可作為前端場地資訊查詢使用。

## 主要流程

- 使用者提交場地預約申請，初始狀態為 `1=審核中`。
- 建立與修改預約時，系統檢查是否與同場地、同日期、已通過預約的時段衝突。
- 使用者可查詢所有個人預約，或依場地、狀態、日期範圍做分頁查詢。
- 使用者可修改 `審核中` 或 `已通過` 的預約，修改後狀態重置為 `審核中`。
- 使用者可撤回 `審核中` 或 `已通過` 的預約，撤回後狀態改為 `0=已撤回`。
- 管理員通過預約時再次做衝突檢查，通過後會自動拒絕其他衝突的審核中申請。
- 管理員可調整預約狀態，從拒絕改為通過時需重新檢查衝突。
- 管理員刪除預約時採軟刪除，狀態改為 `4=已刪除`，並刪除該預約的設備關聯。
- 日曆 API 提供月、周、日三種視圖，協助前端呈現場地占用與個人預約。

## 資料生命週期

預約主資料儲存在 `bookings`。狀態值目前使用整數表示：

- `0`：已撤回。
- `1`：審核中。
- `2`：已通過。
- `3`：已拒絕。
- `4`：已刪除，僅管理端軟刪除使用。

預約時段以 `time_slots` 的 24-bit mask 儲存，每一個 bit 代表一個小時。前端與 API 使用 `slots: List<Integer>`，例如 `[8, 9]` 表示 `08:00-10:00`。

設備借用透過 `booking_equipment` 關聯預約與設備。建立或修改預約時，若 request 帶 `equipmentIds`，後端會寫入對應關聯。

## 衝突規則

- 衝突只以 `status=2` 的已通過預約為阻擋條件。
- `審核中`、`已拒絕`、`已撤回` 不會阻擋使用者送出申請。
- 判斷方式為同場地、同日期且 `(existing.time_slots & requestMask) != 0`。
- 審核通過時若無已通過衝突，會將該預約改為通過，並批量拒絕其他衝突的 `審核中` 預約。

## 介面需求

- 預約表單需提交場地、日期、時段、用途、人數、聯絡資訊與可選設備。
- 個人申請頁需顯示場地名稱、日期、時段、狀態、用途、人數、聯絡資訊與設備名稱清單。
- 個人申請查詢需支援場地、狀態、日期範圍與分頁。
- 日曆頁需支援月視圖的日期摘要、周視圖的每日占用時段、日視圖的詳細個人預約。
- 管理端審核頁需支援場地、日期範圍、狀態篩選，並可查看預約詳情與設備清單。

## 注意事項

- `getMyBookings` 目前使用 `selectByUserId` 後在 service 手動組 VO，場地名稱暫以 `"場地 " + venueId` 顯示，詳情不足；分頁查詢則會透過 SQL join 取得真實場地名稱與設備。
- `updateBooking` 的衝突檢查目前未排除自身，若已通過預約只改其他欄位但時段不變，可能會被自己的原預約判定為衝突。
- 月視圖的 `hasUserBooking` 目前固定為 `false`，尚未依登入使用者資料計算。
- 日視圖的 `venueName` 目前暫以 `"場地 " + venueId` 顯示，尚未查詢真實場地名稱。
