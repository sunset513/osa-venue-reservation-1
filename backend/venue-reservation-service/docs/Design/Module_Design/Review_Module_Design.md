# 審核核心引擎：模組功能描述文件 (V1.0)

**日期：** 2026-04-15
**狀態：** 詳細設計階段

## 一、 模組概觀 (Module Overview)

審核核心引擎是系統中負責預約申請審查的關鍵模組。它負責處理管理員對預約申請的審核流程，包含待審核預約列表管理、審核決策制定、衝突檢查防呆、狀態流轉控制以及軟刪除機制。該模組旨在提供一個嚴謹且透明的審核機制，確保校內場地資源的合理分配與公平利用。

## 二、 核心業務邏輯流程 (Core Business Logic Flow)

當管理員透過 API 對預約申請進行審核時，核心引擎將執行以下流程：

### 2.1 獲取預約列表流程

1. **查詢條件構建**：根據查詢參數（場地ID、日期範圍、狀態）構建查詢條件。
2. **狀態篩選**：
   - 若指定 status 參數，則只查詢該狀態的預約案件。
   - 若不指定 status 參數，則查詢全部狀態的預約案件，但排除已刪除 (status=4) 的案件。
3. **結果轉換**：將資料庫記錄轉換為 `BookingVO` 物件，包含場地名稱、設備清單等信息。
4. **統一回傳**：使用 `Result<List<BookingVO>>` 格式統一回傳結果。

### 2.2 審核通過流程 (Review Approval Flow)

1. **預約案查詢**：根據 bookingId 從資料庫查詢該筆預約案。
2. **狀態驗證**：檢查預約案當前狀態是否為「審核中 (1)」或「拒絕 (3)」，確保只有待審核或可重新審核的案件才能被通過。
3. **衝突檢查（防呆機制）**：
   * 系統會掃描資料庫中同場地、同日期且狀態為「已通過 (2)」的預約案件。
   * 執行位元與運算 (`&`)，檢查時段是否重疊。
   * 若發現衝突，則阻斷審核並回傳「該時段已被其他已通過之申請佔用」的錯誤訊息。
4. **狀態更新**：若通過衝突檢查，將該筆預約案狀態改為「已通過 (2)」。
5. **連鎖拒絕**：自動將同場地、同日期、時段重疊的其他「審核中 (1)」狀態預約案改為「拒絕 (3)」。
6. **事務提交**：所有操作包裹在資料庫事務中，確保原子性。

### 2.3 狀態變更流程 (Status Update Flow)

1. **預約案查詢**：根據 bookingId 查詢該筆預約案。
2. **狀態變更驗證**：根據狀態變更規則，決定是否允許該狀態轉換。
3. **衝突檢查**：如果目標狀態為「已通過 (2)」，則執行衝突檢查。
4. **狀態更新**：更新預約案狀態。
5. **連鎖操作**（視情況）：
   * 若從「已通過 (2)」改為「拒絕 (3)」，無須連鎖操作。
   * 若從「拒絕 (3)」改為「已通過 (2)」，需執行衝突檢查並進行連鎖拒絕。

### 2.4 軟刪除流程 (Soft Delete Flow)

1. **預約案查詢**：根據 bookingId 查詢該筆預約案。
2. **存在性驗證**：確保預約案存在且未被刪除。
3. **狀態更新**：將預約案狀態改為「已刪除 (4)」。
4. **級聯刪除**：刪除該預約案在 `booking_equipment` 表中的相關記錄（設備紀錄）。
5. **事務提交**：所有操作包裹在資料庫事務中。

## 三、 時段遮罩運算規範 (Bitmasking Specification)

審核模組複用 Booking 模組的時段遮罩技術，用於衝突檢查。

### 3.1 定義規則
* **切割單位**：將一天 24 小時分為 **24 個時段 (Slots)**。
* **對應關係**：Slot $n$ 代表時段 $n:00 \sim (n+1):00$。
* **存儲格式**：使用一個 `Unsigned Integer` (32-bit，僅使用低 24 位元)。

### 3.2 衝突判定公式
假設預約 A 的遮罩為 $M_A$，預約 B 的遮罩為 $M_B$，則：
* **無衝突條件**：$M_A \& M_B = 0$（位元與結果為 0）
* **有衝突條件**：$M_A \& M_B \neq 0$（存在重疊時段）

### 3.3 範例參考
| 預約時段 | 選取的 Slots | 十進位遮罩值 | 衝突判定 |
| :--- | :--- | :--- | :--- |
| 08:00 - 09:00 | `[8]` | 256 | 與 09:00 - 10:00 無衝突 |
| 09:00 - 10:00 | `[9]` | 512 | 256 & 512 = 0 ✓ |
| 08:00 - 10:00 | `[8, 9]` | 768 | 與上述均有衝突 |
| 20:00 - 24:00 | `[20, 21, 22, 23]` | 15,728,640 | 用於跨越午夜情況 |

## 四、 併發與資料一致性方案 (Concurrency & Consistency)

審核系統屬於「低併發寫」但「高併發讀」場景。為確保審核結果的準確性和一致性，採取以下措施：

### 4.1 衝突檢查條件
系統僅針對狀態為 **`APPROVED` (2)** 的預約進行衝突鎖定。
* **審核中 (1) 的多筆申請**可以時段重疊。
* **只有當申請通過審核 (2)** 後，其他同場地、同日期、同時段的「審核中 (1)」申請才會被自動拒絕。

### 4.2 樂觀鎖保護 (Optimistic Locking)
在 `bookings` 表中利用 `version` 欄位保護審核操作。
* **用途**：防止管理員在同時審核多筆重疊申請時，發生「多筆申請同時通過」導致衝突的 Race Condition。
* **操作**：執行 Update 時，必須帶上查詢時的 version 值，若版本不符則拋出異常，令操作重試。

### 4.3 事務管理
所有審核操作（包括狀態更新、連鎖拒絕、軟刪除）必須包裹在資料庫事務中，確保以下操作的原子性：
* 主表 (bookings) 的狀態更新。
* 關聯表 (booking_equipment) 的操作。
* 多筆預約的連鎖狀態變更。

### 4.4 事務隔離級別
* **建議使用**：`REPEATABLE_READ`（默認）或 `SERIALIZABLE`（高併發場景下）。
* **理由**：防止髒讀、不可重複讀，確保衝突檢查的準確性。

## 五、 狀態流轉定義 (State Machine)

審核系統中預約單的狀態流轉如下：

### 5.1 基礎狀態定義
| 狀態碼 | 狀態名稱 | 說明 |
| :--- | :--- | :--- |
| **0** | 已撤回 (WITHDRAWN) | 申請人或管理員撤回的申請 |
| **1** | 審核中 (PENDING) | 初始申請狀態，等待管理員審核 |
| **2** | 已通過 (APPROVED) | 管理員核准，預約時段已被鎖定 |
| **3** | 已拒絕 (REJECTED) | 管理員拒絕或被自動拒絕（衝突） |
| **4** | 已刪除 (DELETED) | 軟刪除狀態，保留歷史記錄 |

### 5.2 狀態流轉規則
| 起始狀態 | 目標狀態 | 觸發動作 | 備註 |
| :--- | :--- | :--- | :--- |
| **1 (審核中)** | **2 (已通過)** | 管理員審核通過 | 執行衝突檢查，通過則連鎖拒絕其他衝突申請 |
| **1 (審核中)** | **3 (拒絕)** | 管理員拒絕 | 簡單狀態變更，無須衝突檢查 |
| **1 (審核中)** | **0 (撤回)** | 管理員或申請人撤回 | 允許 |
| **2 (已通過)** | **3 (拒絕)** | 管理員撤銷審核 | 允許，釋放已鎖定的時段 |
| **2 (已通過)** | **0 (撤回)** | 管理員或申請人撤回 | 允許 |
| **3 (拒絕)** | **1 (審核中)** | 管理員重新審核 | 允許，案件重新進入審核隊列 |
| **3 (拒絕)** | **2 (已通過)** | 管理員強制通過 | 執行衝突檢查，通過則連鎖拒絕其他衝突申請 |
| **X (任意狀態)** | **4 (已刪除)** | 管理員刪除 | 軟刪除，保留歷史記錄用於審計 |

## 六、 核心方法設計 (Core Methods Design)

### 6.1 getPendingBookings
**功能**：獲取預約列表

**參數**：
- `venueId` (Long)：場地 ID，預設為 1
- `startDate` (LocalDate)：開始日期，預設為當月初
- `endDate` (LocalDate)：結束日期，預設為當月末
- `status` (Integer)：預約狀態（可選），支援 0-3 的狀態值，不填則查詢全部除了已刪除的 status=4

**業務邏輯**：
1. 構建查詢條件：場地 ID、日期範圍、以及可選的狀態過濾
2. 若指定 status 參數，則只查詢該狀態的預約案；若不指定，則查詢全部狀態（除了 status=4）
3. 查詢所有符合條件的預約案
4. 轉換為 BookingVO 物件（包含設備名稱清單）
5. 回傳 `Result<List<BookingVO>>`

**異常處理**：
- 若查詢結果為空，回傳空列表
- 若參數非法（如日期格式錯誤、狀態值超出範圍），拋出 `IllegalArgumentException`
- 若狀態值不在 0-3 範圍且不為 null，拋出異常並提示「狀態值無效，請輸入 0-3 或不填」

### 6.2 getBookingDetails
**功能**：獲取特定預約案詳細資訊

**參數**：
- `bookingId` (Long)：預約案編號

**業務邏輯**：
1. 根據 bookingId 查詢預約案
2. 若不存在，拋出 `BookingNotFoundException`
3. 關聯查詢設備清單和場地信息
4. 轉換為 BookingVO 物件
5. 回傳 `Result<BookingVO>`

**異常處理**：
- 預約案不存在時，拋出自定義異常 `BookingNotFoundException`

### 6.3 reviewBooking
**功能**：審核預約申請（通過）

**參數**：
- `bookingId` (Long)：預約案編號

**業務邏輯**：
1. 開啟事務
2. 查詢預約案，驗證狀態為 1（審核中）或 3（拒絕）
3. 執行衝突檢查：
   - 查詢同場地、同日期、狀態為 2（已通過）的所有預約
   - 對每筆預約執行時段遮罩與運算
   - 若發現衝突，拋出 `BookingConflictException`
4. 更新預約案狀態為 2（已通過），使用樂觀鎖（version）
5. 連鎖拒絕同場地、同日期、時段重疊的其他「審核中」申請
6. 提交事務
7. 回傳 `Result<Void>`

**異常處理**：
- 預約案不存在：`BookingNotFoundException`
- 衝突檢查失敗：`BookingConflictException`
- 樂觀鎖失敗：`OptimisticLockException`

**涉及多表操作**：使用 `@Transactional(rollbackFor = Exception.class)`

### 6.4 updateReviewStatus
**功能**：更新預約審核狀態

**參數**：
- `bookingId` (Long)：預約案編號
- `newStatus` (Integer)：目標狀態

**業務邏輯**：
1. 開啟事務
2. 查詢預約案
3. 根據當前狀態和目標狀態，執行相應的業務邏輯：
   - **1 → 2**：同 reviewBooking，執行衝突檢查和連鎖拒絕
   - **2 → 3**：簡單狀態變更，無須衝突檢查
   - **3 → 2**：執行衝突檢查和連鎖拒絕
   - **其他**：簡單狀態變更
4. 更新預約案狀態，使用樂觀鎖
5. 提交事務
6. 回傳 `Result<Void>`

**異常處理**：
- 預約案不存在：`BookingNotFoundException`
- 衝突檢查失敗：`BookingConflictException`
- 樂觀鎖失敗：`OptimisticLockException`

### 6.5 deleteBooking
**功能**：刪除預約案（軟刪除）

**參數**：
- `bookingId` (Long)：預約案編號

**業務邏輯**：
1. 開啟事務
2. 查詢預約案
3. 將預約案狀態改為 4（已刪除）
4. 刪除 booking_equipment 表中與該預約相關的所有設備記錄
5. 提交事務
6. 回傳 `Result<Void>`

**異常處理**：
- 預約案不存在：`BookingNotFoundException`

**涉及多表操作**：使用 `@Transactional(rollbackFor = Exception.class)`

## 七、 異常定義 (Exception Hierarchy)

### 7.1 自定義異常類
```
RuntimeException
├── BookingNotFoundException       # 預約案不存在
├── BookingConflictException       # 時段衝突
├── OptimisticLockException        # 樂觀鎖失敗
└── InvalidStatusTransitionException # 非法的狀態轉換
```

### 7.2 異常對應的 HTTP 響應
所有異常由 `GlobalExceptionHandler` 捕獲並轉化為統一的 `Result.error()` 格式，HTTP 狀態碼統一為 **200 OK**（業務邏輯層面的失敗）。

## 八、 VO 複用策略

Review 模組複用 Booking 模組中的 `BookingVO`，作為所有 API 的統一資料傳輸物件。

### 8.1 複用的原因
1. **資料一致性**：避免同一概念的多個定義導致的結構不同步
2. **代碼簡潔性**：減少重複的 VO 類定義
3. **維護成本低**：修改預約資訊格式時只需改一個地方
4. **前端效率高**：統一的資料結構定義，減少適配工作

### 8.2 BookingVO 欄位說明
| 欄位名 | 類型 | 說明 |
| :--- | :--- | :--- |
| `id` | Long | 申請案編號 |
| `venueName` | String | 場地名稱 |
| `bookingDate` | LocalDate | 預約日期 |
| `slots` | List\<Integer\> | 預約時段列表（0-23） |
| `status` | Integer | 申請狀態 |
| `purpose` | String | 使用用途 |
| `pCount` | Integer | 預估人數 |
| `contactInfo` | String | 聯絡人資訊（JSON 字串） |
| `equipments` | List\<String\> | 設備名稱列表 |
| `createdAt` | LocalDateTime | 申請時間 |

## 九、 資料庫設計 (Database Schema)

### 9.1 主要操作的表
- **bookings**：預約申請主表（主要操作對象）
- **booking_equipment**：預約設備關聯表（軟刪除時級聯刪除）
- **venues**：場地資訊表（JOIN 獲取場地名稱）
- **equipments**：設備主表（JOIN 獲取設備名稱）

### 9.2 索引優化
- `idx_venue_date`：(venue_id, booking_date) 複合索引，優化衝突檢查查詢
- `idx_user_id`：申請人查詢索引

### 9.3 樂觀鎖欄位
- `version`：每次更新時遞增，用於防止 Race Condition

## 十、 後續優化方案 (Future Enhancements)

以下功能規劃為未來版本實現，暫不在 MVP 階段包含：

1. **權限控制**：使用 Spring Security 的 `@PreAuthorize` 註解，確保只有管理員角色能執行審核操作。

2. **發信通知**：在審核狀態變更（通過、拒絕、撤銷）時，使用 `JavaMailSender` 發送郵件通知申請人。使用 AOP 分離發信邏輯與核心業務邏輯。

3. **審計日誌**：使用 SLF4J Logger 記錄所有審核操作（操作人員、操作時間、操作內容、操作結果等），便於後續的審計與問題追蹤。

4. **批量操作**：支持管理員批量通過、批量拒絕等操作，提高管理效率。

5. **審核統計**：提供審核週期、通過率、拒絕率等統計數據，幫助優化審核流程。
