# 場地與組織模組：模組功能描述文件 (V1.0)

**日期：** 2026-04-03  
**狀態：** 詳細設計階段

## 一、 模組概觀 (Module Overview)

場地與組織模組負責維護系統的基礎靜態資料，包含管理單位（Units）與場地資訊（Venues）。此模組的主要職責是提供前端渲染選單所需的數據，並在預約流程中提供場地限制（如容納人數）的驗證基準。

## 二、 核心功能 (Core Features)

本模組目前鎖定「學務處」進行開發，但設計上完全兼容多單位擴展：

1.  **管理單位查詢 (Unit Discovery)**：
    * 提供全校可供租借的單位列表。
    * MVP 階段資料庫雖預留 3 個單位空間，但 API 預設僅返回目前啟用的「學務處本部」。
2.  **場地清單檢索 (Venue Listing)**：
    * 使用者選定單位後，系統根據 `unit_id` 過濾並呈現該單位所屬的場地清單。
    * 返回資訊包含場地名稱、容納人數及基本介紹。
3.  **場地詳情與限制查詢 (Venue Metadata)**：
    * 提供單一場地的詳細規則說明（Description）與人數上限（Capacity）。
    * 預約核心引擎會以此數據作為「預估人數」是否符合規範的判定依據。

## 三、 多租戶與擴展設計 (Expansion Strategy)

為了達成「一份代碼服務全校」的軟體工程目標，本模組採用以下擴展策略：

### 1. 組織解耦 (Decoupling)
* **多租戶架構**：透過 `units` 與 `venues` 的一對多關聯，未來只需在 `units` 表增加紀錄（如：住服組、課外組），即可直接開啟新單位的場地服務，無需修改程式碼邏輯。
* **單位代碼識別**：每個單位擁有唯一的 `code`（如：學務處為 `OSA`），便於未來與校方 Portal 進行權限與身分對接。

### 2. 場地屬性封裝
* 場地資訊中包含 `JSON` 格式或 `TEXT` 的擴展欄位（如 `description`），可用於存放不同單位特有的借用規則，保持 Schema 的靈活性。

## 四、 軟體工程思維 (Architectural Decisions)

1.  **高頻讀取優化 (Read-Heavy)**：
    * 場地與單位資料屬於極少變動、頻繁讀取的數據。
    * **設計決策**：未來可引入快取機制（如 Redis），降低資料庫負擔並加速前端選單載入。
2.  **資料完整性限制**：
    * **唯一性約束**：單位名稱與代碼均為 `Unique`，防止管理設定衝突。
    * **級聯刪除 (Cascading)**：當單位被移除時，相關場地資訊應同步處理，確保無孤兒數據。

## 五、 關鍵資料結構 (Data Constraints)

| 組件 | 關鍵欄位 | 約束規則 |
| :--- | :--- | :--- |
| **單位 (Unit)** | `name`, `code` | 必須唯一，不可重複 |
| **場地 (Venue)** | `unit_id` | 必須關聯至有效的單位 ID |
| **場地 (Venue)** | `capacity` | 必須為大於 0 的整數 |
