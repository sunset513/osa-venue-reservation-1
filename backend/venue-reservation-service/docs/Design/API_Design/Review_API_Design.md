# 審核核心引擎：模組 API 文件 (V1.0)

**日期：** 2026-04-15
**基礎路徑：** `/api`
**認證方式：** 於 Header 攜帶 `Authorization: mock-token-123` (MVP 階段使用)

## 一、 數據傳輸物件 (DTO/VO) 設計

在進行 API 呼叫前，請參考以下結構定義：

### 1. ReviewRequestDTO (審核申請提交)
| 欄位名稱 | 類型 | 必填 | 說明 |
| :--- | :--- | :--- | :--- |
| `bookingId` | Long | 是 | 預約申請案編號 |
| `status` | Integer | 是 | 審核狀態 (0:撤回, 1:審核中, 2:通過, 3:拒絕, 4:刪除) |

### 2. BookingVO (預約詳細資訊 - 複用現有 VO)
此 VO 已在 Booking 模組中定義，Review 模組直接複用，無需重新定義。

| 欄位名稱 | 類型 | 說明 |
| :--- | :--- | :--- |
| `id` | Long | 申請案編號 |
| `venueName` | String | 場地名稱 |
| `bookingDate` | LocalDate | 預約日期（ISO 8601 格式：YYYY-MM-DD） |
| `slots` | List\<Integer\> | 預約時段清單（24 小時制索引 0-23） |
| `status` | Integer | 申請狀態 (0:已撤回, 1:審核中, 2:已通過, 3:已拒絕) |
| `purpose` | String | 場地使用用途說明 |
| `pCount` | Integer | 預估人數 |
| `contactInfo` | String | 聯絡人資訊（JSON 格式：包含 name、phone、email） |
| `equipments` | List\<String\> | 所借用的設備名稱清單 |
| `createdAt` | LocalDateTime | 申請時間 |

**備註：** Review 模組複用此 VO 用於：
- 待審核預約列表的回傳物件
- 預約詳細資訊查詢的回傳物件
- 審核狀態變更時的確認資訊

這樣可以保持資料結構一致性，避免前端需要適配多種物件格式。

## 二、 API 終端點 (Endpoints)

### 1. 獲取預約列表
- **方法：** `GET`
- **路徑：** `/reviews/pending`
- **功能：** 根據查詢參數獲取預約案列表，支援按場地、日期範圍及狀態進行過濾。
- **查詢參數：**
  | 參數名稱 | 類型 | 必填 | 說明 |
  | :--- | :--- | :--- | :--- |
  | `venueId` | Long | 否 | 場地ID (預設為1) |
  | `startDate` | LocalDate | 否 | 開始日期 (格式：YYYY-MM-DD，預設為當月初) |
  | `endDate` | LocalDate | 否 | 結束日期 (格式：YYYY-MM-DD，預設為當月末) |
  | `status` | Integer | 否 | 預約狀態 (0:未提交, 1:審核中, 2:已通過, 3:已拒絕，不填則查詢全部除了已刪除的 status=4) |

### 2. 獲取特定預約案詳細資訊
- **方法：** `GET`
- **路徑：** `/reviews/bookings/{id}`
- **功能：** 查詢特定預約案的詳細資訊，供管理員審核使用。
- **路徑參數：**
  | 參數名稱 | 類型 | 說明 |
  | :--- | :--- | :--- |
  | `id` | Long | 預約申請案編號 |

- **響應格式 (`Result<BookingVO>`)：**
    - **成功 (200 OK)：**
      ```json
      {
        "success": true,
        "message": "操作成功",
        "data": {
          "id": 1,
          "venueName": "會議室 A",
          "bookingDate": "2026-04-10",
          "slots": [8, 9],
          "status": 1,
          "purpose": "專案討論",
          "pCount": 5,
      ```

      ```json
      { "success": false, "message": "查詢的預約案不存在", "data": null }
      ```

### 3. 審核預約申請（通過）
- **方法：** `POST`
- **路徑：** `/reviews/bookings/{id}/approve`
- **功能：** 核准預約申請。審核通過前會再次檢查衝突，若通過則自動拒絕其他衝突的同場地、同日期、同時段申請。
- **路徑參數：**
  | 參數名稱 | 類型 | 說明 |
  | :--- | :--- | :--- |
  | `id` | Long | 預約申請案編號 |

- **響應格式 (`Result<Void>`)：**
    - **成功 (200 OK)：**
      ```json
      { "success": true, "message": "操作成功", "data": null }
      ```
    - **業務錯誤 (200 OK)：**
      ```json
      { "success": false, "message": "該時段已被其他已通過之申請佔用", "data": null }
      ```
    - **其他錯誤 (200 OK)：**
      ```json
      { "success": false, "message": "該預約案已被審核，無法重複審核", "data": null }
      ```

### 4. 更新預約審核狀態
- **方法：** `PUT`
- **路徑：** `/reviews/bookings/{id}/status`
- **功能：** 修改審核狀態。支援多種狀態變更，狀態從「拒絕(3)」改為「通過(2)」時會進行衝突檢查。
- **路徑參數：**
  | 參數名稱 | 類型 | 說明 |
  | :--- | :--- | :--- |
  | `id` | Long | 預約申請案編號 |

- **請求範例：**
  ```json
  {
    "bookingId": 10,
    "status": 3
  }
  ```

- **狀態變更規則：**
  | 原狀態 | 目標狀態 | 說明 |
  | :--- | :--- | :--- |
  | 1(審核中) | 3(拒絕) | 允許拒絕審核中的申請 |
  | 2(通過) | 3(拒絕) | 允許撤銷已通過的申請 |
  | 3(拒絕) | 1(審核中) | 允許將拒絕的申請改為審核中 |
  | 3(拒絕) | 2(通過) | 允許，但需進行衝突檢查 |
  | 其他 | 任意 | 允許其他狀態變更，例如 1→0、2→0、0→1、0→2 等 |

- **響應格式 (`Result<Void>`)：**
    - **成功 (200 OK)：**
      ```json
      { "success": true, "message": "操作成功", "data": null }
      ```
    - **業務錯誤 (200 OK)：**
      ```json
      { "success": false, "message": "該時段已被其他已通過之申請佔用", "data": null }
      ```

### 5. 刪除預約案
- **方法：** `DELETE`
- **路徑：** `/reviews/bookings/{id}`
- **功能：** 刪除特定預約案。使用軟刪除方式，將狀態改為「已刪除(4)」，同時刪除相關的設備紀錄。
- **路徑參數：**
  | 參數名稱 | 類型 | 說明 |
  | :--- | :--- | :--- |
  | `id` | Long | 預約申請案編號 |

- **響應格式 (`Result<Void>`)：**
    - **成功 (200 OK)：**
      ```json
      { "success": true, "message": "操作成功", "data": null }
      ```
    - **案件不存在 (200 OK)：**
      ```json
      { "success": false, "message": "查詢的預約案不存在", "data": null }
      ```

## 三、 VO 複用策略說明

### BookingVO 複用原理
`BookingVO` 是 Booking 模組中定義的預約資訊物件，包含了預約的所有必要資訊：
- **基礎資訊**：申請案編號、場地名稱、預約日期、時段
- **審核資訊**：申請狀態（0-3，對應撤回/審核中/已通過/已拒絕）
- **使用用途**：目的、預估人數、聯絡資訊（JSON 字串格式）
- **設備清單**：所借用設備名稱清單
- **時間戳**：申請時間（createdAt）

Review 模組無需重新定義詳細資訊物件，直接複用 `BookingVO`，這樣可以：
1. **保持資料一致性**：使用同一個對象定義，避免資料結構不同步
2. **簡化維護**：減少重複代碼，修改時只需在一個地方進行
3. **提高效率**：前端可以統一理解同一個數據結構，無論來自 Booking 或 Review 模組

### 待審核列表的簡化展示
雖然 API 返回 `BookingVO`，但前端可根據實際需求選擇性顯示欄位。例如待審核列表可能只需要以下欄位：
- `id` - 申請案編號
- `venueName` - 場地名稱
- `bookingDate` - 預約日期
- `slots` - 預約時段清單
- `purpose` - 使用用途
- `status` - 狀態 (1:審核中)
- `pCount` - 預估人數
- `createdAt` - 申請時間

## 四、 錯誤處理機制 (Error Handling)

當系統發生異常時，會由 `GlobalExceptionHandler` 捕獲並轉化為 `Result.error()` 格式：

1. **參數校驗失敗 (`400 Bad Request`)**：若請求參數不符規範（如 bookingId 為負數、狀態值超出範圍），回傳校驗錯誤訊息。

2. **時段衝突 (`Booking Conflict`)**：業務邏輯拋出 `RuntimeException`，顯示「該時段已被其他已通過之申請佔用」。

3. **預約案不存在**：當查詢或操作的預約案不存在時，回傳「查詢的預約案不存在」的錯誤訊息。

4. **狀態變更非法**：當預約案狀態不符合操作要求時，回傳相應的業務錯誤訊息。

5. **未預期錯誤**：回傳「伺服器內部錯誤，請稍後再試」。

## 五、 業務規則說明

### 審核流程
- 預約申請初始狀態為「審核中(1)」，管理員可透過 API 進行審核。
- 通過審核後，狀態變更為「通過(2)」，且其他衝突申請會自動改為「拒絕(3)」。
- 已通過的申請可被管理員改為「拒絕(3)」或重新進行審核。

### 衝突檢查
- 任何通過審核的申請或狀態變更為「通過(2)」的申請，都會再次進行衝突檢查。
- 衝突定義：同一場地、同一日期、時段重疊的已通過申請。
- 若發現衝突，不允許該申請通過審核或狀態變更。

### 軟刪除
- 刪除預約案時，不直接從資料庫移除，而是將狀態改為「已刪除(4)」並刪除相關的設備紀錄。
- 軟刪除可保留歷史紀錄用於審計。
