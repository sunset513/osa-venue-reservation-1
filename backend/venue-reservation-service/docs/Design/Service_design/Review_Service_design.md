# 審核核心引擎：模組 Service 設計文件 (V1.0)

**日期：** 2026-04-15  
**類別：** `tw.edu.ncu.osa.venue_reservation_service.service.ReviewService`

## 一、 設計概述

`Service` 層遵循 **單一職責原則 (SRP)**，僅處理與預約相關的業務邏輯。為了提升系統的健壯性，我們將利用 Spring 的聲明式事務（`@Transactional`）確保多表操作的原子性，並透過拋出特定的 `RuntimeException` 配合 `GlobalExceptionHandler` 來達成統一的錯誤響應格式。

## 二、 核心方法說明
### 1. 獲取待審核預約列表 (`getPendingBookings`)
* 功能描述：查詢所有狀態為的預約案，供管理員審核使用。
* 參數：場地ID (預設為1)、日期範圍 (預設為當月)。
* 詳細功能細節
* 1. 基本上複用類似 getMyBookings 的邏輯，但不只要查出單一用戶的預約案，而是查詢特定場地所有的預約案。
* 2. 查詢條件需加入場地ID，可以參考 BookingMapper 中的 getBookingsByVenueAndDateRange、selectBookingsByDateRangeForCalendar 等方法，進行修改。

### 2. 審核預約申請 (`reviewBooking`)

* 功能描述：審核預約申請，同時檢查審核通過的時段是否已經衝突(防呆)，同時相同時段的申請會被拒絕。
* 參數：選定欲通過審核的該筆Booking的id。
* 詳細功能細節：
* 1. 通過的審核要將該筆預約案的狀態改為「已通過 (2)」
* 2. 通過審核前再次進行衝突檢查，檢查是否有其他同場地、同日期、同時段的預約案是「已通過 (2)」，如果有就不能通過審核，要顯示「該時段已被其他已通過之申請佔用」的錯誤訊息。
* 3. 若審核通過，則將同場地、同日期、同時段的其他預約案狀態改為「拒絕 (3)」。

### 3. 審核狀態修改 (`updateReviewStatus`)
* 功能描述：修改審核狀態，例如從「審核中(1)」改為「拒絕(3)」。
* 參數：選定欲修改的該筆Booking的id
* 詳細功能細節：
* 1. 此端點保持彈性，可以將任何預約的狀態改為任意狀態，但狀態從1(審核中)改為2(審核通過)的功能，主要使用reviewBooking。
* 2. 具體狀態如下
* 0:撤回, 1:審核中, 2:通過, 3:拒絕
* 1 -> 3 審核中的申請可以拒絕
* 2 -> 3 已通過的申請可以改為拒絕
* 3 -> 1 拒絕的申請可以改為審核中
* 3 -> 2 拒絕的申請可以改為審核通過，但要再次檢查是否有衝突
* 3. 其他狀態變更不受限制，例如 1 -> 0、2 -> 0、0 -> 1、0 -> 2 等等。但通常已撤回的申請不會再被修改為審核中或已通過，除非管理員有特殊需求。

### 4. 獲取特定預約案的詳細資訊 (`getBookingDetails`)
* 功能描述：查詢特定預約案的詳細資訊，供管理員審核使用。
* 參數：Booking的id
* 詳細功能細節：
* 1. 查詢該筆預約案的詳細資訊，包括場地、日期、時段、使用用途、預估人數、聯絡資訊等。
* 2. 可以參考 BookingMapper 中的 getBookingById 方法，進行修改。


### 5. 刪除預約案 (`deleteBooking`)
* 功能描述：刪除特定預約案，供管理員使用。
* 參數：Booking的id
* 詳細功能細節：
* 1. 刪除該筆預約案，並且同時刪除該筆預約案相關的設備紀錄（booking_equipment表中的紀錄）。
* 2. 使用軟刪除的方式，將該筆預約案的狀態改為「已刪除 (4)」，而不是直接從資料庫中刪除，以保留歷史紀錄。

### 共通原則
* DB 操作 : 使用 ReviewMapper 進行資料庫操作，確保 SQL 語句的優化與安全性。可以參考目前的 BookingMapper 的實作，進行類似的 SQL 編寫。
* 統一響應 : 使用 `Result` 類統一 API 的回應格式，成功時回傳 `Result.success(data)`，失敗時回傳 `Result.error(message)`。是否需要回傳資料視具體情況而定，例如 `getPendingBookings` 會回傳預約列表，而 `reviewBooking` 可能只回傳操作結果訊息。
* 異常處理 : 在 Service 層直接拋出 `RuntimeException`，由全局異常攔截器統一處理，確保錯誤訊息的一致性與安全性。具體異常類型可以根據業務需求進行細分，例如 `BookingNotFoundException`、`BookingConflictException` 等等，以提供更具體的錯誤訊息給前端。
* 事務管理 : 對於涉及多表操作的方法（如 `reviewBooking`），使用 `@Transactional(rollbackFor = Exception.class)` 確保操作的原子性，避免資料不一致的情況發生。

### 後續優化(先不做，留待未來版本)
* 權限控制 : 在 Service 層加入權限檢查，確保只有具有管理員角色的用戶才能執行審核相關操作。可以使用 Spring Security 的 `@PreAuthorize` 註解來實現。
* 發信通知 : 在審核狀態變更（如審核通過、拒絕）時，加入發信通知功能，讓申請人能即時收到審核結果的通知。使用 Spring 的 `JavaMailSender` 來實現郵件發送。並且使用 AOP 技術將發信邏輯與核心業務邏輯分離，保持 Service 層的清晰與專注。
* 日誌記錄 : 在關鍵操作（如審核通過、拒絕、刪除）中加入日誌記錄，方便後續的審計與問題追蹤。可以使用 SLF4J 的 Logger 來記錄操作細節，例如操作人員、操作時間、操作結果等資訊。
